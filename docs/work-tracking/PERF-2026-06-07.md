# Performance — Diagnosis + First Fix (2026-06-07)

User: "this needs a damned speed up — it's always been ridiculously slow."

## Diagnosis (measured on the real Android device)
- Page **shell** loads fast: `domContentLoaded` ≈ 0.6s.
- But **stats took 11.5s** to appear (dashes meanwhile).
- **Network is NOT the bottleneck**: when warm, all data calls total ~3s
  (slowest `getPlayerStatsFiltered` 679ms, `getConversations` 510ms, etc.).
- The ~8–11s gap is **non-network critical-path gating**:
  1. The whole identity render is `await`ed behind `waitForAuthReady()` (up to 5s),
     then `getPlayerSession` (+ a 600ms retry), then `getPlayerStatsFiltered` — all **serial**.
  2. **Cold starts**: functions are **gen-1 with NO `minInstances`** (447 exports, zero warm-keeping),
     so the first call of each after idle adds 2–5s.
  3. The "optimistic cache" block existed but called `renderIdentity(null, {})` → it showed
     **dashes, not cached stats**.

## Fix shipped (frontend, safe) — v73 / home-vnext.js v55
**Optimistic identity cache (stale-while-revalidate) on Home.** Cache last-known
{player, stats, rank, record, challenges} to **localStorage**; on every open render it
**instantly**, then let the fresh (cold-prone) calls resolve in the background and swap in.

### Measured result (real device, cached load)
- Time-to-stats: **11,482ms → 358ms** (~32× faster). Stats appear as fast as the page paints.

## Recommended next levers (biggest remaining wins)
1. **Eliminate cold starts — `minInstances: 1` on the hot functions**
   (`getPlayerSession`, `getPlayerStatsFiltered`, `getConversations`, `getPlayerChallenges`,
   `getUnreadNotificationCount`, the triples-snapshot reader, `getActiveOnlineMatches`).
   This is THE systemic fix for "always slow" — but it keeps instances warm 24/7, which adds a
   small always-on Cloud Functions cost. **Your call** (needs a backend deploy + billing OK).
   Alternative with no standing cost: **batch** Home's 5 calls into one `getHomeBundle` function.
2. **Extend the optimistic-cache pattern** to the Up Next card (persist the triples snapshot to
   localStorage too) and to Profile / Triples / Clubhouse — same instant-render trick.
3. **Parallelize** the remaining serial `await`s (session → stats → snapshot don't all need to be
   sequential).
4. **Skeleton loaders** instead of "-" dashes for any still-loading bits (feels faster, less "broken").

## Reusable device-perf tooling (kept)
- `scripts/qa/droid-net.mjs` — network waterfall on the device (slowest data calls).
- `scripts/qa/droid-eval.mjs` — run JS on the device tab via CDP.

## Rollout (v74/v75) — optimistic cache across the app + measured wins

Extended the proven Home pattern to Profile, Triples, Clubhouse + Home's Up Next snapshot
(parallel subagents for the 3 pages, verified on the real Android device).

| Page | Before (cold, mobile) | After (cached) | Win |
|------|----------------------|----------------|-----|
| Home — stats | 11,482 ms | **358 ms** | ~32× |
| Triples — standings | ~10 s | **572 ms** | ~17× |
| Profile | ~10 s (cold) | instant from cache | ✓ |
| Clubhouse — hero stats | cold | instant from cache | ✓ |

Cache keys: `brdc:vnext:home:identity:v1`, `brdc:vnext:home:snapshot:<league>:v1`,
`brdc:vnext:profile:self:v1`, `brdc:vnext:triples:snapshot:v1`, `brdc:vnext:clubhouse:v1`.

### Gotcha fixed during rollout
The Triples (and Home) snapshot caches initially **failed silently** — the league `matches`
carry full throw-by-throw `games[]` arrays (multi-MB) → `QuotaExceededError` on the localStorage
write (swallowed by try/catch). Fixed by **slimming** the cached snapshot: drop `games`/`throws`/
`lineups`/`sets` per match (and `statsById` on Home, since stats come from the identity cache).
Triples cache dropped to **284 KB** and writes cleanly. Lesson: anything cached to localStorage
must exclude the heavy per-match game detail.

### Still the systemic lever (your call)
The optimistic cache makes repeat opens instant, but the FIRST-ever open (empty cache) still eats
cold starts. To kill those: `minInstances: 1` on the ~7 hot functions (standing cost) OR a batched
`getHomeBundle` (no standing cost). Recommend the batch.

## getHomeBundle — batched function (v76, functions deployed to brdc-v2)

Built the batched read so Home makes **1 cold-startable round-trip instead of 3**:
- New `getHomeBundle` (gen-1 onRequest) returns `{ player, stats, challenges }` in one call.
- Extracted the cores of the 3 existing functions into shared helpers
  (`resolveFirebaseSessionPlayer`, `computeFilteredStats`, `computePlayerChallenges`) so logic is
  single-source — the existing HTTP handlers were refactored to call them, **response shapes
  unchanged**.
- Home calls `getHomeBundle` first (session+stats+challenges in parallel server-side), with a
  **fallback** to the 3 legacy calls if the bundle ever fails. `loadChallengesForBadge` reuses the
  bundle's challenges (no extra call).

### Verified on the real device (authed)
- `getHomeBundle` → `success:true`, player "Donnie Pagel", stats (raw aggregates correct), challenges. ✓
- Home renders 3DA 52.6 / MPR 2.48 / Rank #6 / Challenges 1. ✓
- **Regression checks** (functions were refactored): `getPlayerStatsFiltered` ✓ (Profile renders 52.6),
  `getPlayerChallenges` ✓. Both response shapes byte-identical.

Net cold-open path: 3 serial cold-startable calls → **1**. Combined with the optimistic cache
(repeat opens instant), Home now loads fast cold AND warm. Functions live on **brdc-v2**;
deploy cmd: `firebase deploy --only functions:getHomeBundle,getPlayerSession,getPlayerStatsFiltered,getPlayerChallenges --project brdc-v2`.

### Optional further step (not done)
`minInstances:1` on `getHomeBundle` alone would make even the first-ever open instant for a tiny
standing cost (1 warm instance instead of 7). Cheapest possible warm-keeping. Your call.

### minInstances:1 on getHomeBundle — DEPLOYED (--force, accepts the standing cost)
`exports.getHomeBundle = functions.runWith({ minInstances: 1 }).https.onRequest(...)`.
One warm instance always running → the first-ever / first-of-the-day open of Home skips the cold
start entirely. Verified warm latency ~377–622ms (vs 2–5s cold). Standing cost: 1 always-on
gen-1 instance. Deploy needs `--force` (Firebase blocks bill-increasing minInstances otherwise).
